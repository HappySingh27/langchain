# Retrievers

A **Retriever** is a component in **LangChain** that fetches relevant documents from a data source in response to a user's query.

# Key Points

- All retrievers in **LangChain** are **Runnables**.
- Every retriever returns a list of **LangChain `Document` objects**.
- Different retrievers solve different retrieval problems:
  - **Wikipedia Retriever** → Retrieves information from Wikipedia.
  - **Vector Store Retriever** → Retrieves semantically similar documents from a vector database.
  - **ArXiv Retriever** → Retrieves research papers from ArXiv.
  - **MMR Retriever** → Retrieves relevant and diverse documents.
  - **Multi-Query Retriever** → Retrieves documents using multiple LLM-generated query variations.
  - **Contextual Compression Retriever** → Retrieves relevant documents and compresses them to only the useful content.

## Types of Retrievers

Retrievers can be classified based on the **data source** or the **search strategy** they use.

---

# 1. Based on Data Source

## a. Wikipedia Retriever

A **Wikipedia Retriever** queries the **Wikipedia API** to fetch relevant content for a given query.

### How It Works

1. You provide a query (e.g., `"Albert Einstein"`).
2. It sends the query to the Wikipedia API.
3. It retrieves the most relevant articles.
4. It returns the results as LangChain `Document` objects.

---

## b. Vector Store Retriever

A **Vector Store Retriever** is the **most common type of retriever** in LangChain. It searches and retrieves documents from a **vector store** based on **semantic similarity** using vector embeddings.

### How It Works

1. Store your documents in a **vector store** (e.g., FAISS, Chroma, Weaviate).
2. Convert each document into a **dense vector** using an **embedding model**.
3. When the user enters a query:
   - The query is also converted into a vector.
   - The retriever compares the query vector with the stored document vectors.
   - It retrieves the **top-k most similar** documents.

---

## c. ArXiv Retriever

An **ArXiv Retriever** retrieves research papers from the **ArXiv** repository based on the user's query.

---

# 2. Based on Search Strategy

## a. MMR (Maximum Marginal Relevance) Retriever

**Maximum Marginal Relevance (MMR)** is an information retrieval algorithm that retrieves documents that are **not only relevant to the query but also different from each other**.

Its goal is to **reduce redundancy** in the retrieved results while maintaining **high relevance** to the user's query.

### Why MMR Retriever?

In a regular similarity search, the retrieved documents may:

- Be very similar to each other.
- Repeat the same information.
- Lack diverse perspectives.

MMR avoids these problems by:

1. Selecting the **most relevant** document first.
2. Selecting the next document that is:
   - Highly relevant to the query.
   - **Least similar** to the already selected documents.
3. Repeating the process until the required number of documents is retrieved.

### Why is MMR useful in RAG?

MMR is especially useful in **Retrieval-Augmented Generation (RAG)** because:

- It provides **diverse but relevant** context.
- It reduces duplicate information.
- It improves answer quality when documents are semantically similar.

### Example

**Query:**

```text
Benefits of Artificial Intelligence
```

**Regular Similarity Search**

1. AI benefits in healthcare.
2. AI benefits in healthcare (another source).
3. AI benefits in healthcare (third source).

**MMR Retrieval**

1. AI benefits in healthcare.
2. AI benefits in education.
3. AI benefits in finance.

This gives the LLM more diverse context while keeping the retrieved documents relevant.

## b. Multi-Query Retriever

A **Multi-Query Retriever** improves retrieval by generating **multiple semantically different versions** of the user's query using an LLM. This helps retrieve documents that a single query might miss.

### Why do we need it?

Sometimes a single query cannot capture all the different ways information is written in the documents.

For example, consider the query:

```text
How can I stay healthy?
```

This query could also mean:

- What should I eat?
- How often should I exercise?
- How can I manage stress?

A simple similarity search may miss documents discussing these topics if they don't explicitly contain the word **"healthy."**

### Example Queries Generated by the LLM

For the query:

```text
How can I stay healthy?
```

The LLM may generate:

1. What are the best foods to maintain good health?
2. How often should I exercise to stay fit?
3. What lifestyle habits improve mental and physical wellness?
4. How can I boost my immune system naturally?
5. What daily routines support long-term health?

Each generated query is then used to search the vector store.

### How It Works

1. Takes the original user query.
2. Uses an **LLM** to generate multiple semantically different queries.
3. Performs retrieval for each generated query.
4. Combines all retrieved documents.
5. Removes duplicate documents.
6. Returns the final set of unique and relevant documents.

### Workflow

```text
                 Original Query
                       │
                       ▼
               +---------------+
               |      LLM      |
               +---------------+
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
    Query 1         Query 2         Query 3 ...
       │               │               │
       └───────────────┼───────────────┘
                       ▼
                 Vector Store
                       │
                       ▼
         Combine + Deduplicate Results
                       │
                       ▼
          Final Retrieved Documents
```

### Advantages

- Improves recall by searching with multiple query variations.
- Finds documents using different wording or synonyms.
- Retrieves documents that a single similarity search may miss.
- Improves the quality of RAG pipelines.

## c. Contextual Compression Retriever

A **Contextual Compression Retriever** is an advanced LangChain retriever that improves retrieval quality by **compressing retrieved documents**, keeping **only the content relevant to the user's query**.

Unlike a normal retriever, it does **not** return the entire retrieved document. Instead, it removes unnecessary information before passing the context to the LLM.

### Example

**Query**

```text
What is photosynthesis?
```

**Retrieved document (Traditional Retriever)**

```text
The Grand Canyon is a famous natural site.
Photosynthesis is how plants convert light into energy.
Many tourists visit every year.
```

### Problem

A traditional retriever returns the **entire paragraph** even though only one sentence is relevant.

Problems include:

- Returns the complete paragraph.
- Only one sentence is actually useful.
- Remaining text is irrelevant noise.
- Wastes the LLM's context window.
- Can reduce answer quality.

### What Contextual Compression Retriever Does

Instead of returning the full paragraph, it returns only the relevant part.

```text
Photosynthesis is how plants convert light into energy.
```

### How It Works

1. A base retriever (e.g., FAISS or Chroma) retrieves **N documents**.
2. A **compressor** (usually an LLM) processes each document.
3. The compressor keeps only the content relevant to the query.
4. Irrelevant content is discarded.
5. The compressed documents are returned.

### Workflow

```text
              User Query
                    │
                    ▼
          Base Retriever (FAISS/Chroma)
                    │
                    ▼
          Retrieved Documents
                    │
                    ▼
        LLM Compressor / Filter
                    │
                    ▼
      Relevant Content Only
                    │
                    ▼
                  LLM
```

### When to Use

Use a Contextual Compression Retriever when:

- Documents are long and contain mixed information.
- You want to reduce the context length sent to the LLM.
- You want to improve answer accuracy in RAG pipelines.
- You want to reduce irrelevant tokens and lower inference cost.

# Other Useful LangChain Retrievers

Besides the commonly used retrievers, LangChain also provides several specialized retrievers.

| Retriever | Purpose |
|-----------|---------|
| **BM25Retriever** | Traditional keyword-based retrieval using the BM25 ranking algorithm. |
| **ParentDocumentRetriever** | Retrieves small chunks but returns their parent document to preserve context. |
| **SelfQueryRetriever** | Uses an LLM to generate structured metadata filters along with semantic search. |
| **TimeWeightedVectorRetriever** | Gives higher priority to recently added documents while still considering similarity. |
| **EnsembleRetriever** | Combines results from multiple retrievers (e.g., BM25 + Vector Search) to improve retrieval quality. |
| **MultiVectorRetriever** | Stores multiple embeddings for the same document to improve retrieval from different perspectives. |
| **ArxivRetriever** | Retrieves research papers directly from the ArXiv repository. |
